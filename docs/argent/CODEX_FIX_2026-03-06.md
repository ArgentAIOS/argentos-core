# OpenAI Codex Fix — 2026-03-06

## What Broke

OpenAI Codex (gpt-5.3-codex) stopped returning real responses. Instead, every request came back as a Cloudflare 502 Bad Gateway HTML page (~153 chars), which the gateway treated as successful content. The dashboard showed either garbled output or "terminated."

This happened right after OpenAI launched GPT-5.4 on March 5-6, 2026. Anthropic and MiniMax providers were unaffected.

## Root Cause: Wrong `baseUrl` Override

Pi's upstream codex provider (`openai-codex-responses.js`) correctly hardcodes:

```
DEFAULT_CODEX_BASE_URL = "https://chatgpt.com/backend-api"
```

It appends `/codex/responses` to this base URL, producing the correct endpoint:

```
https://chatgpt.com/backend-api/codex/responses
```

**The problem:** Our `~/.argentos/provider-registry.json` had the openai-codex provider's `baseUrl` set to `https://api.openai.com/v1`. This is the standard OpenAI API endpoint — NOT the Codex endpoint.

### The Override Chain

On every gateway startup, this chain runs:

1. `provider-registry.json` (user-editable source of truth)
2. `ensureArgentModelsJson()` in `src/agents/models-config.ts` reads from provider registry
3. Generates `~/.argentos/agents/main/agent/models.json` (merge strategy: generated wins)
4. Pi's model registry loads `models.json`, overriding its own `models.generated.js` defaults
5. `resolveCodexUrl(model.baseUrl)` builds: `https://api.openai.com/v1/codex/responses`
6. That endpoint doesn't exist → Cloudflare 502

This meant even manually fixing `models.json` was useless — it got overwritten on every gateway restart from the wrong value in `provider-registry.json`.

## What We Fixed

### Fix 1: Correct the baseUrl (provider-registry.json)

Changed `openai-codex.baseUrl` from `https://api.openai.com/v1` to `https://chatgpt.com/backend-api` in `~/.argentos/provider-registry.json`. This is the true source of truth that feeds `models.json` on startup.

### Fix 2: Inject OAuth JWT into Pi's streamSimple (attempt.ts)

Pi's codex provider needs an `apiKey` in stream options. It checks:

```js
const apiKey = options?.apiKey || getEnvApiKey(model.provider) || "";
```

But `getEnvApiKey("openai-codex")` has no env var mapping, and our auth comes from OAuth (JWT stored in `authStorage.runtimeOverrides`). Without injection, the API key is empty and the request fails.

**File:** `src/agents/pi-embedded-runner/run/attempt.ts` (~line 787)

Before: Pi's `streamSimple` was called without the API key for openai-codex.

After:

```typescript
if (params.provider === "openai-codex") {
  const runtimeOverrides = (params.authStorage as any).runtimeOverrides as
    | Map<string, string>
    | undefined;
  const codexApiKey = runtimeOverrides?.get("openai-codex");
  if (codexApiKey) {
    activeSession.agent.streamFn = (model, context, options) =>
      streamSimple(model, context, { ...options, apiKey: codexApiKey, transport: "websocket" });
  } else {
    activeSession.agent.streamFn = streamSimple;
  }
}
```

This wraps Pi's own `streamSimple` (which correctly dispatches to `streamSimpleOpenAICodexResponses` via the api registry) and injects:

- `apiKey`: The OAuth JWT from `runtimeOverrides`
- `transport`: WebSocket (see below)

### Fix 3: Added GPT-5.4 and GPT-5.2 models (provider-registry.json)

GPT-5.4 launched March 5-6 but wasn't in Pi's npm package or our registry. Added to `openai-codex` provider:

| Model   | Context   | Max Output | Reasoning | Vision | Input $/1M | Output $/1M |
| ------- | --------- | ---------- | --------- | ------ | ---------- | ----------- |
| gpt-5.4 | 1,050,000 | 128,000    | yes       | yes    | $2.50      | $15.00      |
| gpt-5.2 | 272,000   | 128,000    | yes       | yes    | $1.75      | $14.00      |

Full model list now: gpt-5.4, gpt-5.3-codex, gpt-5.3-codex-spark, gpt-5.2-codex, gpt-5.2, gpt-5.1-codex-max, gpt-5.1-codex-mini.

### Fix 4: WebSocket transport (attempt.ts)

Changed from default SSE to WebSocket transport for all openai-codex requests.

Pi's codex provider supports two streaming transports:

- **SSE** (default): New HTTP connection per request. TLS handshake + headers every time.
- **WebSocket**: Persistent connection, reused across turns. Pi caches the socket per session with a 5-minute idle TTL. Falls back to SSE automatically if WebSocket fails.

WebSocket is exclusive to the Codex endpoint (`chatgpt.com/backend-api`) — not available on standard OpenAI API, Anthropic, or other providers.

## What We Did NOT Change

- No changes to the OAuth flow or auth profiles
- No changes to Pi's upstream codex provider code
- No changes to the Argent runtime or other providers
- No changes to the dashboard
- Pi's `streamSimple` remains the underlying dispatch — we just wrap it with the missing options

## Auth Flow (unchanged, for reference)

1. `resolveApiKeyForProvider({provider: "openai-codex"})` finds the `openai-codex:default` OAuth profile
2. JWT not expired → `buildOAuthApiKey()` returns `credentials.access`
3. `authStorage.setRuntimeApiKey("openai-codex", jwt)` stores it in `runtimeOverrides` Map
4. Our wrapper reads it from `runtimeOverrides.get("openai-codex")` and passes as `options.apiKey`
5. Pi's codex provider sends `Authorization: Bearer <jwt>` + extracts `chatgpt_account_id` from JWT claims

## Verification

Confirmed working 2026-03-06:

- gpt-5.3-codex: responds correctly over WebSocket
- gpt-5.4: responds correctly over WebSocket
- Gateway log shows: `[openai-codex] Using Pi streamSimple with injected OAuth JWT (websocket transport)`

## Lessons

1. **`provider-registry.json` is the true source of truth** for model baseUrls, not `models.json`. The latter is regenerated on every gateway startup.
2. **Pi's codex provider is solid** — the correct endpoint, retry logic, reasoning support, and WebSocket transport are all built in. We just need to feed it the right baseUrl and apiKey.
3. **WebSocket transport is Codex-only** — an OpenAI beta feature (`responses_websockets=2026-02-06`). Other providers (Anthropic, Google, MiniMax) are SSE-only.
